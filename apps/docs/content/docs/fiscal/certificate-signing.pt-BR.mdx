## Visão Geral

Notas fiscais eletrônicas brasileiras exigem assinatura digital com certificado A1 e-CNPJ (formato PFX/PKCS#12). O módulo trata a extração PFX, exibição de informações do certificado e assinatura digital XML seguindo a especificação NF-e MOC 4.00.

**Arquivos**: `certificate.ts`, `sefaz-transport.ts` (reutiliza a extração de certificado)

## Tratamento PFX

### Por que openssl?

O crypto nativo do Bun não consegue parsear arquivos PKCS#12 (PFX) — o módulo `pkcs12` não está implementado. A solução alternativa usa o CLI `openssl` via `child_process`:

```bash
# Extract certificate (PEM)
openssl pkcs12 -in cert.pfx -clcerts -nokeys -passin pass:xxx -legacy

# Extract private key (PEM)
openssl pkcs12 -in cert.pfx -nocerts -nodes -passin pass:xxx -legacy
```

A flag `-legacy` é necessária para arquivos PFX antigos que usam criptografia legada (RC2-40).

### extractCertFromPfx / extractKeyFromPfx

```typescript
export function extractCertFromPfx(pfxBuffer: Buffer, passphrase: string): string
export function extractKeyFromPfx(pfxBuffer: Buffer, passphrase: string): string
```

Ambas:
1. Gravam o buffer PFX em um arquivo temporário
2. Executam `openssl pkcs12` para extrair o PEM
3. Parseiam a saída com regex para isolar o bloco PEM (`-----BEGIN CERTIFICATE-----...-----END CERTIFICATE-----`)
4. Limpam o arquivo temporário no `finally`
5. Lançam erro se o bloco PEM não for encontrado

Essas funções são exportadas e compartilhadas com `sefaz-transport.ts` (que precisa dos arquivos PEM para mTLS com curl).

### loadCertificate

```typescript
export function loadCertificate(pfx: Buffer, passphrase: string): CertificateData
```

Retorna `{ privateKey, certificate, pfxBuffer, passphrase }` — as strings PEM necessárias para assinatura e o PFX bruto para armazenamento.

### getCertificateInfo

```typescript
export function getCertificateInfo(pfx: Buffer, passphrase: string): CertificateInfo
```

Extrai detalhes legíveis do certificado para a UI de configurações:
- `commonName` — nome da empresa no certificado
- `validFrom`, `validUntil` — datas de validade
- `serialNumber` — número serial do certificado
- `issuer` — nome da AC (Autoridade Certificadora)

Usa `node:crypto.X509Certificate` (disponível no Bun).

## Assinatura XML

### signXml

```typescript
export function signXml(xml: string, privateKeyPem: string, certificatePem: string): string
```

Assina um documento XML de NF-e conforme a especificação de assinatura MOC 4.00:

1. **Referência**: O elemento `<infNFe Id="NFe...">` é a referência assinada
2. **Canonicalização**: C14N exclusiva (`http://www.w3.org/2001/10/xml-exc-c14n#`)
3. **Transformações**:
   - Assinatura envelopada (remove `<Signature>` antes do hash)
   - C14N exclusiva
4. **Digest**: SHA-1 (`http://www.w3.org/2000/09/xmldsig#sha1`)
5. **Algoritmo de assinatura**: RSA-SHA1 (`http://www.w3.org/2000/09/xmldsig#rsa-sha1`)
6. **KeyInfo**: Contém `<X509Data><X509Certificate>` com certificado Base64 (sem cabeçalhos)

### Posicionamento da assinatura

O elemento `<Signature>` é inserido dentro de `<NFe>`, após `<infNFe>`:

```xml
<NFe xmlns="http://www.portalfiscal.inf.br/nfe">
  <infNFe Id="NFe35..." versao="4.00">
    ...
  </infNFe>
  <Signature xmlns="http://www.w3.org/2000/09/xmldsig#">
    <SignedInfo>
      <CanonicalizationMethod Algorithm="..." />
      <SignatureMethod Algorithm="..." />
      <Reference URI="#NFe35...">
        <Transforms>...</Transforms>
        <DigestMethod Algorithm="..." />
        <DigestValue>...</DigestValue>
      </Reference>
    </SignedInfo>
    <SignatureValue>...</SignatureValue>
    <KeyInfo>
      <X509Data>
        <X509Certificate>MIIGxTCCBa...</X509Certificate>
      </X509Data>
    </KeyInfo>
  </Signature>
</NFe>
```

### Biblioteca

Usa [xml-crypto](https://github.com/node-saml/xml-crypto) para a operação de assinatura propriamente dita. A biblioteca trata canonicalização, hashing e computação da assinatura.

## Notas de segurança

- Senhas PFX são armazenadas criptografadas no banco de dados (campo `certificate_password`)
- Arquivos temporários são sempre limpos em blocos `finally`
- Chaves privadas PEM nunca são persistidas em disco além da operação de assinatura
- A expiração do certificado é rastreada em `certificate_valid_until` para alertas na UI
