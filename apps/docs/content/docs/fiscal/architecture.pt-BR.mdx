## Visão Geral

O módulo fiscal implementa a emissão completa de documentos fiscais eletrônicos brasileiros (NF-e modelo 55 e NFC-e modelo 65) seguindo a especificação SEFAZ MOC 4.00. Foi portado da biblioteca PHP [sped-nfe](https://github.com/nfephp-org/sped-nfe) para TypeScript, adaptado a uma arquitetura em camadas DDD (Domain-Driven Design).

### Estrutura do Monorepo

O projeto utiliza um monorepo Turborepo. A lógica fiscal pura está em um pacote independente (`@finopenpos/fiscal`) em `packages/fiscal/`, sem dependências de banco de dados. Os arquivos acoplados ao banco (repositórios e serviço de notas) ficam na camada da aplicação em `apps/web/src/lib/`.

- **`packages/fiscal/src/`** — pacote fiscal independente (lógica de domínio, XML, SEFAZ, etc.)
- **`apps/web/src/lib/`** — arquivos de serviço/repositório acoplados ao banco (invoice-service, invoice-repository, fiscal-settings-repository)
- **`apps/web/src/`** — aplicação Next.js (páginas, routers tRPC, etc.)

## Fluxo de Dependências

```
invoice-service
├── fiscal-settings-repository ──→ DB (fiscalSettings)
├── invoice-repository ──→ DB (invoices, invoiceItems, invoiceEvents)
├── certificate.ts ──→ node:crypto, xml-crypto
├── xml-builder.ts
│   ├── tax-icms.ts ──→ tax-element.ts
│   ├── tax-pis-cofins-ipi.ts ──→ tax-element.ts
│   ├── value-objects/ (AccessKey, TaxId)
│   ├── xml-utils.ts (tag, escapeXml)
│   └── format-utils.ts
├── sefaz-client.ts
│   ├── sefaz-transport.ts ──→ curl (mTLS)
│   ├── sefaz-request-builders.ts
│   ├── sefaz-response-parsers.ts
│   └── sefaz-urls.ts
├── complement.ts (protocol attachment)
└── qrcode.ts (NFC-e QR code)
```

**Regra chave**: Módulos de impostos (domínio) nunca importam xml-builder (infraestrutura). Eles retornam estruturas `TaxElement`, que o xml-builder serializa. Isso evita dependências circulares e mantém a lógica de domínio pura.

## Convenções Numéricas

| Conceito | Armazenamento | Exemplo | Saída XML |
|----------|---------------|---------|-----------|
| Monetário (vBC, vICMS, vNF) | Centavos (inteiro) | `1050` | `"10.50"` |
| Alíquotas ICMS/FCP (pICMS, pFCP) | Centésimos (4dp) | `180000` | `"18.0000"` |
| Alíquotas PIS/COFINS (pPIS) | x10000 (4dp) | `16500` | `"1.6500"` |
| Quantidades | x1000 (3dp) | `1500` | `"1.500"` |

## Tipos de Domínio Principais

| Tipo | Valores | Significado |
|------|---------|-------------|
| `InvoiceModel` | `55`, `65` | NF-e (B2B), NFC-e (consumidor) |
| `InvoiceStatus` | `pending`, `authorized`, `rejected`, `cancelled`, `denied`, `contingency`, `voided` | Estados do ciclo de vida |
| `SefazEnvironment` | `1`, `2` | Produção, Homologação |
| `EmissionType` | `1`, `6`, `7`, `9` | Normal, SVC-AN, SVC-RS, Offline |
| `TaxRegime` | `1`, `2`, `3` | Simples Nacional, Excesso Simples, Normal |

## Multi-Tenancy

Todas as tabelas do banco são chaveadas por `user_uid`. Toda função de repositório recebe `userUid` como primeiro parâmetro. Não há acesso cruzado entre tenants.
