## Visão Geral

O motor de impostos calcula os tributos brasileiros para cada item da nota fiscal e produz objetos `TaxElement` estruturados que o construtor XML serializa. Esse desacoplamento (a lógica de domínio nunca manipula XML diretamente) é o padrão DDD central do módulo fiscal.

**Arquivos**: `tax-element.ts`, `tax-icms.ts`, `tax-pis-cofins-ipi.ts`, `tax-issqn.ts`, `tax-is.ts`, `format-utils.ts`

## O Padrão TaxElement

```
tax-icms.ts ──→ TaxElement ──→ xml-builder.ts ──→ XML string
tax-pis-cofins-ipi.ts ─┘            ↑
                            serializeTaxElement()
```

### Estrutura TaxElement (`tax-element.ts`)

```typescript
interface TaxField {
  name: string;   // XML tag name, e.g. "vBC"
  value: string;  // formatted value, e.g. "10.50"
}

interface TaxElement {
  outerTag: string | null;  // wrapping tag, e.g. "ICMS", "PIS"
  outerFields: TaxField[];  // fields on the outer tag (e.g. IPI's cEnq)
  variantTag: string;       // variant tag, e.g. "ICMS00", "PISAliq"
  fields: TaxField[];       // fields inside the variant
}
```

**Serialização**: `<ICMS><ICMS00><orig>0</orig><CST>00</CST>...</ICMS00></ICMS>`

### Helpers de campos

- `requiredField(name, value)` — sempre incluído (lança erro se null em modo estrito)
- `optionalField(name, value)` — retorna `null` se o valor for null/undefined
- `filterFields(fields)` — remove entradas null do array

## ICMS (`tax-icms.ts`)

O imposto mais complexo — 15 variantes CST (regime Normal) + 10 variantes CSOSN (Simples Nacional).

### Ponto de entrada

```typescript
calculateIcms(data: IcmsData): { element: TaxElement; totals: IcmsTotals }
```

Roteia por `taxRegime`:
- `1` ou `2` (Simples Nacional) → variantes CSOSN (101, 102, 201, 202, 500, 900)
- `3` (Normal) → variantes CST (00, 02, 10, 15, 20, 30, 40, 51, 53, 60, 61, 70, 90)

### Acumulação de totais

Cada função CST acumula valores em `IcmsTotals` (vBC, vICMS, vBCST, vST, vFCP, vFCPST, etc.). Esses totais são agregados entre todos os itens e usados em `<total><ICMSTot>`.

### Helpers de blocos de campos de domínio (padrão DRY)

Conceitos fiscais comuns extraídos como helpers:

```typescript
fcpFields(d)           // FCP: vBCFCP, pFCP, vFCP (5 CSTs)
fcpStFields(d)         // FCP-ST: vBCFCPST, pFCPST, vFCPST (7 variants)
stFields(d)            // ST base: modBCST … vICMSST (6 variants, required)
desonerationFields(d)  // vICMSDeson, motDesICMS, indDeduzDeson (5 CSTs)
stDesonerationFields(d) // vICMSSTDeson, motDesICMSST (3 CSTs)
```

### Construtores especiais

- `buildIcmsPartXml(data)` — ICMSPart (partição interestadual)
- `buildIcmsStXml(data)` — ICMSST (repasse ST)
- `buildIcmsUfDestXml(data)` — ICMSUFDest (DIFAL, consumidor final interestadual)

## PIS / COFINS / IPI / II (`tax-pis-cofins-ipi.ts`)

### Padrão ContributionTax (DRY)

PIS e COFINS são estruturalmente idênticos — apenas os nomes dos campos diferem. Um `ContributionTaxConfig` genérico parametriza:

```typescript
interface ContributionTaxConfig {
  taxName: string;      // "PIS" | "COFINS"
  rateField: string;    // "pPIS" | "pCOFINS"
  valueField: string;   // "vPIS" | "vCOFINS"
  stTag: string;        // "PISST" | "COFINSST"
  stIndicator: string;  // "indSomaPISST" | "indSomaCOFINSST"
}
```

Adaptadores públicos finos mapeiam tipos específicos para o motor genérico:

```typescript
export function calculatePis(data: PisData): TaxElement {
  return calculateContributionTax(
    { CST: data.CST, vBC: data.vBC, rate: data.pPIS, value: data.vPIS, ... },
    PIS_CONFIG,
  );
}
```

### Roteamento por CST

Tanto PIS quanto COFINS roteiam por CST:
- `01`, `02` → Aliq (base + alíquota + valor)
- `03` → Qtde (baseado em quantidade)
- `04`-`09` → NT (não tributado, apenas CST)
- `49`-`99` → Outr (outros, campos condicionais)

### IPI

- `IPITrib` (CST 00, 49, 50, 99): dois métodos de cálculo (vBC x pIPI ou qUnid x vUnid)
- `IPINT` (demais CSTs): apenas campo CST
- Campos externos: CNPJProd, cSelo, qSelo, cEnq (código de enquadramento legal)

### II (Imposto de Importação)

Elemento simples de 4 campos: vBC, vDespAdu, vII, vIOF.

## Utilitários de Formatação (`format-utils.ts`)

| Função | Entrada | Saída | Caso de uso |
|--------|---------|-------|-------------|
| `formatCents(1050)` | int centavos | `"10.50"` | Valores monetários |
| `formatCents(1050, 4)` | int centavos | `"10.5000"` | Alíquotas 4dp |
| `formatRate4(16500)` | int x10000 | `"1.6500"` | Alíquotas PIS/COFINS |
| `formatCentsOrZero(null)` | nullable | `"0.00"` | Campos obrigatórios |
| `formatCentsOrNull(null)` | nullable | `null` | Campos opcionais |

## Como o construtor XML usa os módulos de impostos

```typescript
// Inside xml-builder.ts buildDet()
const { xml: icmsXml, totals } = buildIcmsXml(item.icmsData);
const pisXml = buildPisXml(item.pisData);
const cofinsXml = buildCofinsXml(item.cofinsData);
// ... compose into <imposto> tag
```

O construtor XML nunca constrói o XML de impostos diretamente — ele sempre delega para os módulos de impostos do domínio.
